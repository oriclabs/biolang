Biostatistics in Practice

Chapter 13: Dimensionality Reduction

The Problem

You have a single-cell RNA-seq dataset: 2,700 peripheral blood mononuclear cells, each measured across roughly 20,000 genes. You want to visualize which cells are similar to each other. You want to discover cell types. You want to understand what gene programs drive the differences.

But you cannot scatter-plot 20,000 dimensions. You cannot compute distances meaningfully in a space this large. And most of those 20,000 genes are noise — housekeeping genes expressed at the same level in every cell, or dropout zeros where the measurement failed entirely.

How do you reduce 20,000 dimensions to 2 or 3 while preserving the meaningful biological structure?

This is the dimensionality reduction problem, and it is the gateway to every single-cell analysis, every bulk transcriptomics comparison, every clinical genomics quality check. Get it wrong, and you will see patterns that do not exist. Get it right, and entire cell atlases emerge from the noise.

Why It Matters

Dimensionality reduction is not just visualization. It is a fundamental analytical step that determines the success or failure of downstream analysis:

In single-cell RNA-seq, PCA followed by UMAP is the standard pipeline. Every cell atlas — the Human Cell Atlas, Tabula Sapiens, the Mouse Cell Atlas — depends on these methods to organize millions of cells into interpretable structures. If your PCA captures batch effects instead of biology, every cluster you find downstream is wrong.

In bulk RNA-seq, PCA is the primary quality control tool. Before running any differential expression analysis, you plot PC1 vs PC2. If samples do not group by biological condition — if instead they group by sequencing lane, RNA extraction date, or technician — you have a batch effect that must be addressed before any statistical test is valid.

In GWAS, PCA on genotype data reveals population structure. Failing to correct for population structure in a case-control study produces thousands of false positive associations. The top principal components of genotype data are routinely included as covariates in every GWAS regression model.

In clinical genomics, dimensionality reduction on tumor expression profiles reveals molecular subtypes that predict treatment response. The PAM50 breast cancer classifier, used clinically worldwide, is fundamentally a projection of expression data onto a low-dimensional subtype space.

The Data

Dataset 1: PBMC 3k (single-cell RNA-seq)

Exploration

Understanding the Curse of Dimensionality

Before applying any method, we need to understand why high-dimensional data is fundamentally different from low-dimensional data.

In high dimensions, distances become meaningless. In a 20,000-dimensional space, the ratio of the farthest point to the nearest point approaches 1 as dimensionality increases. Every point is approximately equidistant from every other point. This means that nearest-neighbor methods, distance-based clustering, and any visualization that relies on preserving distances will fail if applied directly to the raw gene expression matrix.

# Demonstrate the curse of dimensionality
# Compute pairwise distances in the full gene space

let all_genes = columns(pbmc) |> filter(|c| c != "barcode")

# Sample 100 cells for illustration
let sample_cells = sample(pbmc, 100)

# Extract expression values for all genes
let expr_matrix = sample_cells |> select(all_genes)

# Compute distance matrix in the full space
let full_dist = dist_matrix(expr_matrix, "euclidean")

# Extract all pairwise distances
let distances = []
let n = len(full_dist)
let i = 0
while i < n do
  let j = i + 1
  while j < n do
    distances = concat(distances, [full_dist[i][j]])
    j = j + 1
  end
  i = i + 1
end

# Look at the distribution of distances
let dist_stats = {
  mean: mean(distances),
  sd: stdev(distances),
  min: min(distances),
  max: max(distances),
  cv: stdev(distances) / mean(distances)
}

println("Distance statistics in full 20,000-D space:")
println("  Mean distance: " + str(round(dist_stats.mean, 2)))
println("  SD of distances: " + str(round(dist_stats.sd, 2)))
println("  CV (SD/mean): " + str(round(dist_stats.cv, 4)))
println("  Min/Max ratio: " + str(round(dist_stats.min / dist_stats.max, 4)))

# The CV will be very small — distances are concentrated
# This means "nearest neighbor" is barely distinguishable from "farthest neighbor"

Looking at Variance Per Gene

Not all genes are informative. Most genes are either not expressed, expressed at a constant level, or dominated by technical noise. The first step in any dimensionality reduction is to identify the genes that carry biological signal.

# Compute variance for each gene across cells
let gene_names = columns(pbmc) |> filter(|c| c != "barcode")
let gene_variances = gene_names |> map(|g| {
  name: g,
  variance: variance(col_values(pbmc, g)),
  mean_expr: mean(col_values(pbmc, g))
})

# Sort by variance, descending
let sorted_genes = gene_variances |> sort(|a, b| b.variance - a.variance)

# Top 20 most variable genes
println("Top 20 most variable genes:")
sorted_genes |> head(20) |> each(|g|
  println("  " + g.name + ": var=" + str(round(g.variance, 2)) +
          " mean=" + str(round(g.mean_expr, 2)))
)

# How many genes have meaningful variance?
let high_var = sorted_genes |> filter(|g| g.variance > 1.0)
println("Genes with variance > 1.0: " + str(len(high_var)))
println("That is " + str(round(100.0 * len(high_var) / len(sorted_genes), 1)) + "% of all genes")

The typical result: fewer than 2,000 genes (10% of the genome) carry most of the variance. The rest are noise. This is why dimensionality reduction works — the data actually lives on a much lower-dimensional manifold.

The Mean-Variance Relationship

In RNA-seq data, variance scales with the mean. Highly expressed genes have higher absolute variance simply because they have more counts. This is a problem for PCA, which finds directions of maximum variance — it would be dominated by highly expressed genes regardless of their biological relevance.

# Plot mean vs variance to see the relationship
let mean_var = gene_variances |> map(|g| {
  log_mean: log10(g.mean_expr + 0.01),
  log_var: log10(g.variance + 0.01)
})

let mv_table = table(mean_var)
scatter(mv_table, "log_mean", "log_var",
  title: "Mean-Variance Relationship (PBMC 3k)",
  x_label: "log10(Mean Expression)",
  y_label: "log10(Variance)")
  |> save_plot("mean_variance_relationship.svg")

# This is why we normalize and select highly variable genes (HVGs)
# before PCA — to remove the mean-variance confound

Normalization: Preparing for PCA

Before PCA, we must normalize the data. The standard single-cell preprocessing pipeline is: (1) normalize to total counts per cell, (2) log-transform, (3) select highly variable genes.

# Step 1: Library-size normalization
# Scale each cell to the same total count (e.g., 10,000)
let target_sum = 10000.0

let normalized = pbmc |> mutate(|row| {
  let total = gene_names |> map(|g| row[g]) |> sum()
  let scale = target_sum / total
  let result = { barcode: row.barcode }
  gene_names |> each(|g|
    result[g] = row[g] * scale
  )
  result
})

# Step 2: Log-transform (log1p to handle zeros)
let log_normalized = normalized |> mutate(|row| {
  let result = { barcode: row.barcode }
  gene_names |> each(|g|
    result[g] = log(1.0 + row[g])
  )
  result
})

# Step 3: Select highly variable genes (top 2000)
let hvg_variances = gene_names |> map(|g| {
  name: g,
  variance: variance(col_values(log_normalized, g))
})

let hvg_sorted = hvg_variances |> sort(|a, b| b.variance - a.variance)
let hvgs = hvg_sorted |> head(2000) |> map(|g| g.name)

println("Selected " + str(len(hvgs)) + " highly variable genes for PCA")

# Step 4: Center and scale (z-score each gene)
let scaled = log_normalized |> select(concat(["barcode"], hvgs))
let scaled_data = normalize(scaled |> select(hvgs), "zscore")

println("Prepared expression matrix: " + str(len(scaled_data)) + " cells x " +
        str(ncols(scaled_data)) + " genes")

The Analysis

PCA: Finding the Axes of Maximum Variance

Principal Component Analysis finds orthogonal directions (principal components) that capture the maximum variance in the data. Each PC is a linear combination of the original genes. The first PC captures the most variance, the second captures the most variance orthogonal to the first, and so on.

# Run PCA on the scaled expression matrix
let pca_result = pca(scaled_data, n_components: 50)

# pca_result contains:
#   .components   — the PC coordinates for each cell (n_cells x n_components)
#   .variance     — variance explained by each component
#   .loadings     — gene weights for each component (n_genes x n_components)

println("PCA complete. Top 10 components — variance explained:")
let total_var = sum(pca_result.variance)
let cumulative = 0.0

pca_result.variance |> head(10) |> each(|v|
  let i = 0
  let pct = 100.0 * v / total_var
  cumulative = cumulative + pct
  println("  PC" + str(i + 1) + ": " + str(round(pct, 2)) + "% (cumulative: " +
          str(round(cumulative, 2)) + "%)")
  i = i + 1
)

What PCA Actually Does — The Mathematics

PCA is an eigenvalue decomposition. For a centered data matrix X (cells x genes), PCA computes the covariance matrix C = X^T X / (n-1), then finds its eigenvectors and eigenvalues. The eigenvectors are the principal component directions. The eigenvalues are the variance explained by each direction.

But you don’t need to compute the full covariance matrix (which would be 20,000 x 20,000). The singular value decomposition (SVD) of X gives the same result more efficiently: X = U S V^T, where the columns of V are the PC loadings (eigenvectors of the covariance matrix) and the diagonal of S squared gives the eigenvalues.

# BioLang's pca() uses SVD internally
# We can verify by computing SVD directly on a small example

let small_data = scaled_data |> head(100) |> select(hvgs |> head(50))
let small_matrix = matrix(small_data)

let svd_result = svd(small_matrix)
# svd_result.u — left singular vectors (cell space)
# svd_result.s — singular values
# svd_result.v — right singular vectors (gene space / loadings)

# Eigenvalues from SVD
let eigenvalues = svd_result.s |> map(|s| s * s / (len(small_matrix) - 1))

println("Top 5 eigenvalues from direct SVD:")
eigenvalues |> head(5) |> each(|e|
  println("  " + str(round(e, 4)))
)

Choosing the Number of PCs: The Elbow Plot

How many PCs should you keep? There is no single correct answer, but the elbow plot provides guidance. Plot the variance explained by each PC. Look for the “elbow” — the point where the curve flattens, indicating that additional PCs are capturing noise rather than signal.

# Create the elbow plot (scree plot)
let var_explained = pca_result.variance |> head(50)
let var_pct = var_explained |> map(|v| 100.0 * v / total_var)

let elbow_data = var_pct |> map(|v|
  let i = 0
  let result = { pc: i + 1, variance_pct: v }
  i = i + 1
  result
)

let elbow_table = table(elbow_data)
scatter(elbow_table, "pc", "variance_pct",
  title: "Scree Plot — Variance Explained by PC",
  x_label: "Principal Component",
  y_label: "% Variance Explained")
  |> save_plot("scree_plot.svg")

# Also plot cumulative variance
let cum_var = []
let running = 0.0
var_pct |> each(|v|
  running = running + v
  cum_var = concat(cum_var, [running])
)

let cum_data = cum_var |> map(|v|
  let i = 0
  let result = { pc: i + 1, cumulative_pct: v }
  i = i + 1
  result
)

let cum_table = table(cum_data)
scatter(cum_table, "pc", "cumulative_pct",
  title: "Cumulative Variance Explained",
  x_label: "Principal Component",
  y_label: "Cumulative % Variance")
  |> save_plot("cumulative_variance.svg")

# Typical single-cell: 10-30 PCs capture most biological signal
# A common heuristic: keep PCs until cumulative variance reaches 80-90%
# or until the scree plot elbow

In the PBMC 3k dataset, you will typically see an elbow around PC 10-15. The first 2-3 PCs separate major cell types (T cells from monocytes from B cells). PCs 4-15 capture finer distinctions (CD4+ vs CD8+ T cells, classical vs non-classical monocytes). Beyond PC 20, you are mostly fitting noise.

Interpreting PC Loadings — What Genes Drive Each Component

Each principal component is a linear combination of genes. The loadings (weights) tell you which genes contribute most to each PC. These loadings have biological meaning — they represent gene programs.

# Get the loadings for PC1 and PC2
# Loadings: which genes have the highest weights in each PC

let pc1_loadings = hvgs |> map(|g|
  let i = 0  # index into hvgs
  { gene: g, loading: pca_result.loadings[i][0] }
)

let pc2_loadings = hvgs |> map(|g|
  let i = 0
  { gene: g, loading: pca_result.loadings[i][1] }
)

# Sort by absolute loading to find the driving genes
let pc1_sorted = pc1_loadings |> sort(|a, b| abs(b.loading) - abs(a.loading))
let pc2_sorted = pc2_loadings |> sort(|a, b| abs(b.loading) - abs(a.loading))

println("Top 10 genes driving PC1:")
pc1_sorted |> head(10) |> each(|g|
  println("  " + g.gene + ": " + str(round(g.loading, 4)))
)

println("Top 10 genes driving PC2:")
pc2_sorted |> head(10) |> each(|g|
  println("  " + g.gene + ": " + str(round(g.loading, 4)))
)

# In PBMC data, PC1 typically separates lymphoid vs myeloid cells
# Key PC1 genes: LYZ, S100A9 (myeloid markers) vs NKG7, GNLY (lymphoid)
# PC2 often separates T cells from B cells and NK cells

The PC1 vs PC2 scatter plot is the most common visualization in genomics. It reveals the global structure of your data at a glance.

t-SNE: Preserving Local Neighborhoods

t-distributed Stochastic Neighbor Embedding (t-SNE) is a nonlinear method that excels at preserving local structure. Unlike PCA, it does not seek directions of maximum variance. Instead, it tries to place cells that are close in the high-dimensional space close together in 2D.

The algorithm:

  1. Compute pairwise similarities in high-D using Gaussian kernels
  2. Define pairwise similarities in 2D using a Student’s t-distribution
  3. Minimize the KL divergence between the two distributions

The t-distribution in step 2 (hence the name) has heavy tails, which helps prevent the “crowding problem” — in low dimensions, moderately distant points would all pile up in a narrow shell if Gaussian kernels were used.

# Run t-SNE on the PCA-reduced data (not on raw expression!)
# This is critical: t-SNE on 20,000 raw genes is slow and noisy
# Standard practice: PCA to 30-50 dims, then t-SNE on PCs

let pc_input = pca_result.components |> map(|row| row |> head(30))
let tsne_default = tsne(pc_input, perplexity: 30)

let tsne_table = table(tsne_default |> map(|row| {
  tSNE1: row[0],
  tSNE2: row[1]
}))

scatter(tsne_table, "tSNE1", "tSNE2",
  title: "PBMC 3k — t-SNE (perplexity=30)",
  x_label: "t-SNE 1",
  y_label: "t-SNE 2")
  |> save_plot("pbmc_tsne_default.svg")

The Perplexity Parameter: When t-SNE Lies

The perplexity parameter controls how many neighbors each point considers. It roughly corresponds to the expected number of neighbors. Low perplexity emphasizes very local structure; high perplexity captures more global patterns.

This is where t-SNE lies. Different perplexity values produce dramatically different plots. Cluster sizes and distances between clusters in a t-SNE plot are NOT meaningful. A cluster that appears twice as large as another may contain the same number of cells. Two clusters that appear far apart may be more similar than two clusters that appear close.

# Run t-SNE with different perplexity values to see the effect
let perplexities = [5, 15, 30, 50, 100]

perplexities |> each(|p|
  let result = tsne(pc_input, perplexity: p)
  let result_table = table(result |> map(|row| {
    tSNE1: row[0],
    tSNE2: row[1]
  }))

  scatter(result_table, "tSNE1", "tSNE2",
    title: "t-SNE perplexity=" + str(p),
    x_label: "t-SNE 1",
    y_label: "t-SNE 2")
    |> save_plot("tsne_perplexity_" + str(p) + ".svg")

  println("Perplexity " + str(p) + " complete")
)

# With perplexity=5: many tiny, tight clusters — overemphasis on local structure
# With perplexity=30: balanced view, standard choice
# With perplexity=100: clusters merge, global structure dominates
# ALL of these are valid — which is why t-SNE is dangerous for quantitative claims

UMAP: The Modern Alternative

Uniform Manifold Approximation and Projection (UMAP) has largely replaced t-SNE as the default visualization method in single-cell genomics. It is faster, better preserves global structure, and produces more reproducible layouts.

UMAP constructs a topological representation of the high-dimensional data using simplicial complexes, then finds a low-dimensional layout that preserves this topological structure. In practice, this means UMAP better preserves the relationships between clusters — clusters that are biologically related tend to appear near each other.

# Run UMAP on the PCA-reduced data
let umap_result = umap(pc_input, n_neighbors: 15, min_dist: 0.5)

let umap_table = table(umap_result |> map(|row| {
  UMAP1: row[0],
  UMAP2: row[1]
}))

scatter(umap_table, "UMAP1", "UMAP2",
  title: "PBMC 3k — UMAP",
  x_label: "UMAP 1",
  y_label: "UMAP 2")
  |> save_plot("pbmc_umap.svg")

UMAP Parameters: n_neighbors and min_dist

Like t-SNE’s perplexity, UMAP has parameters that affect the visualization. Understanding them prevents misinterpretation.

n_neighbors controls how many neighbors define the local structure. Low values (5-15) emphasize fine local structure. High values (50-200) capture broader patterns. Default: 15.

min_dist controls how tightly points are packed. Low values (0.0-0.1) produce tighter clusters with more empty space between them. High values (0.5-1.0) produce a more uniform spread. Default: 0.1.

# Compare UMAP with different parameter settings
let umap_configs = [
  { nn: 5,  md: 0.1, label: "local-tight" },
  { nn: 15, md: 0.1, label: "default-tight" },
  { nn: 15, md: 0.5, label: "default-spread" },
  { nn: 50, md: 0.1, label: "global-tight" },
  { nn: 50, md: 0.5, label: "global-spread" }
]

umap_configs |> each(|cfg|
  let result = umap(pc_input, n_neighbors: cfg.nn, min_dist: cfg.md)
  let result_table = table(result |> map(|row| {
    UMAP1: row[0],
    UMAP2: row[1]
  }))

  scatter(result_table, "UMAP1", "UMAP2",
    title: "UMAP n_neighbors=" + str(cfg.nn) + " min_dist=" + str(cfg.md),
    x_label: "UMAP 1",
    y_label: "UMAP 2")
    |> save_plot("umap_" + cfg.label + ".svg")
)

# Key observation: all of these show the same major clusters
# but with very different visual appearance
# UMAP is more robust than t-SNE to parameter choice, but still varies

PCA vs t-SNE vs UMAP: When to Use Each

Each method has strengths and appropriate use cases:

MethodPreservesSpeedReproducibleBest For
PCAGlobal variance structureFastYes (deterministic)QC, batch detection, pre-processing
t-SNELocal neighborhoodsSlowNo (stochastic)Publication figures with clear clusters
UMAPLocal + some globalFastMostly (seed-dependent)Exploratory analysis, large datasets

Use PCA when: you need quantitative interpretation (loadings, variance explained), you are checking for batch effects, you need a deterministic result, or you are preparing input for downstream methods.

Use t-SNE when: you want to visualize clearly separated clusters in publication figures, and you understand that distances between clusters are not meaningful.

Use UMAP when: you are doing exploratory analysis, working with large datasets (>100k cells), or need to preserve relationships between clusters.

# Extract marker gene expression

# Create combined table with all three embeddings

  let i = 0
  let result = {

    PC2: row[1],

# t-SNE shows them as tight, well-separated clusters
# UMAP shows them as connected but distinct islands

PCA on GTEx: Tissues Separate by Biology

Now we apply PCA to the GTEx bulk RNA-seq data. This is a fundamentally different use case — instead of thousands of cells, we have tens of tissues, each summarized by a single median expression vector.

# Prepare GTEx data for PCA
# Columns are tissues, rows are genes
# We need to transpose: make tissues the observations

let tissue_names = columns(gtex) |> filter(|c| c != "Name" and c != "Description")

println("GTEx tissues: " + str(len(tissue_names)))

# Select the top 5000 most variable genes across tissues
let gene_vars = []
let gene_ids = col_values(gtex, "Name")
let gene_symbols = col_values(gtex, "Description")

tissue_names |> head(1) |> each(|t|
  println("Example tissue column: " + t)
)

# Compute variance across tissues for each gene
let var_per_gene = gene_ids |> map(|gid|
  let row_idx = 0  # would need to find the row for this gene
  let values = tissue_names |> map(|t| col_values(gtex, t)[row_idx])
  { gene: gid, variance: variance(values) }
)

# Select top 5000 most variable genes
let top_genes = var_per_gene |> sort(|a, b| b.variance - a.variance) |> head(5000)
let selected_gene_ids = top_genes |> map(|g| g.gene)

# Build tissue x gene matrix
let tissue_data = tissue_names |> map(|t|
  let values = col_values(gtex, t)
  let result = { tissue: t }
  selected_gene_ids |> each(|gid|
    let idx = 0  # index of this gene
    result[gid] = log2(values[idx] + 1.0)
  )
  result
)

let tissue_table = table(tissue_data)

# Run PCA
let gene_cols = selected_gene_ids
let tissue_expr = tissue_table |> select(gene_cols)
let tissue_normalized = normalize(tissue_expr, "zscore")
let tissue_pca = pca(tissue_normalized, n_components: 10)

# Plot PC1 vs PC2 with tissue labels
let tissue_pc_table = table(tissue_names |> map(|t|
  let i = 0
  { tissue: t, PC1: tissue_pca.components[i][0], PC2: tissue_pca.components[i][1] }
))

scatter(tissue_pc_table, "PC1", "PC2",
  title: "GTEx — PCA of Tissue Expression Profiles",
  x_label: "PC1",
  y_label: "PC2")
  |> save_plot("gtex_tissue_pca.svg")

println("GTEx PCA complete")
println("PC1 likely separates brain tissues from all others")

In the GTEx data, PCA reveals beautiful biological structure:

  • Brain tissues cluster tightly together, separated from all other tissues on PC1
  • Testis is an extreme outlier on PC2 (unique transcriptional program)
  • Blood and immune tissues cluster together
  • Muscle tissues (skeletal, cardiac) cluster together
  • Liver stands somewhat alone (unique metabolic program)

This is PCA working as a biological discovery tool. Nobody told the algorithm which tissues are similar — it recovered known biology purely from expression patterns.

PCA for Batch Effect Detection

One of the most critical applications of PCA in genomics is detecting batch effects. If your PC1 separates by sequencing lane rather than by biological condition, you have a problem that no downstream test can fix.

# Simulated batch effect detection example
# Suppose we have expression data with sample metadata

let expr_data = read_tsv("expression_with_batches.tsv")
let metadata = read_tsv("sample_metadata.tsv")

# Run PCA on expression
let expr_cols = columns(expr_data) |> filter(|c| c != "sample_id")
let expr_matrix = expr_data |> select(expr_cols)
let expr_scaled = normalize(expr_matrix, "zscore")
let batch_pca = pca(expr_scaled, n_components: 10)

# Add metadata to PC coordinates
let pc_with_meta = table(metadata |> map(|m|
  let i = 0
  {
    sample: m.sample_id,
    condition: m.condition,      # biological variable of interest
    batch: m.sequencing_batch,   # technical variable
    PC1: batch_pca.components[i][0],
    PC2: batch_pca.components[i][1]
  }
))

# Test: does PC1 correlate with batch or condition?
let pc1_values = col_values(pc_with_meta, "PC1")

# Group PC1 by condition
let cond_groups = pc_with_meta |> group_by("condition")
let cond_a = cond_groups |> filter(|g| g.key == "treatment") |> map(|g| g.PC1)
let cond_b = cond_groups |> filter(|g| g.key == "control") |> map(|g| g.PC1)
let cond_test = ttest(cond_a, cond_b)

# Group PC1 by batch
let batch_groups = pc_with_meta |> group_by("batch")
let batch_1 = batch_groups |> filter(|g| g.key == "batch1") |> map(|g| g.PC1)
let batch_2 = batch_groups |> filter(|g| g.key == "batch2") |> map(|g| g.PC1)
let batch_test = ttest(batch_1, batch_2)

println("PC1 association with condition: p=" + str(round(cond_test.pvalue, 4)))
println("PC1 association with batch: p=" + str(round(batch_test.pvalue, 4)))

# If batch_test.pvalue << cond_test.pvalue, you have a batch effect
# The data must be corrected before differential expression

PCA as Pre-processing: The Standard Pipeline

In single-cell genomics, PCA is not the final step. It is the pre-processing step before clustering, UMAP visualization, and trajectory inference. The standard pipeline is:

  1. Normalize and log-transform
  2. Select highly variable genes
  3. Scale (z-score)
  4. PCA to 30-50 dimensions
  5. Use PCs (not raw genes) for all downstream analysis

This is because: (a) PCA denoises the data by discarding low-variance components, (b) distances in PC space are more meaningful than in gene space, (c) it is computationally efficient — downstream algorithms work on 30-50 dimensions instead of 20,000.

# The standard single-cell pipeline in BioLang

# Step 1-3: Already done above (normalize, log, HVG selection, scaling)

# Step 4: PCA to 30 dimensions
let n_pcs = 30
let final_pca = pca(scaled_data, n_components: n_pcs)
let pc_embedding = final_pca.components |> map(|row| row |> head(n_pcs))

println("PC embedding: " + str(len(pc_embedding)) + " cells x " +
        str(n_pcs) + " dimensions")

# Step 5a: Cluster in PC space (covered in Chapter 14)
let clusters = leiden(pc_embedding, resolution: 1.0)
println("Found " + str(len(unique(clusters))) + " clusters")

# Step 5b: UMAP for visualization, using PC embedding as input
let final_umap = umap(pc_embedding, n_neighbors: 15, min_dist: 0.1)

# Combine everything
let cell_data = table(pbmc |> map(|row|
  let i = 0
  {
    barcode: row.barcode,
    cluster: clusters[i],
    UMAP1: final_umap[i][0],
    UMAP2: final_umap[i][1],
    PC1: pc_embedding[i][0],
    PC2: pc_embedding[i][1]
  }
))

# Plot UMAP colored by cluster
scatter(cell_data, "UMAP1", "UMAP2",
  color: "cluster",
  title: "PBMC 3k — UMAP colored by Leiden cluster",
  x_label: "UMAP 1",
  y_label: "UMAP 2")
  |> save_plot("pbmc_umap_clusters.svg")

# Count cells per cluster
let cluster_counts = cell_data |> count_by("cluster")
println("Cells per cluster:")
cluster_counts |> each(|c|
  println("  Cluster " + str(c.cluster) + ": " + str(c.n) + " cells")
)

Annotating PCs with Known Biology

Once we have clusters and PC embeddings, we can ask: which genes define each cluster? And do the PCs correspond to known biological programs?

# For each cluster, find the top marker genes
# A marker gene is one that is highly expressed in one cluster vs all others

let cluster_ids = unique(clusters) |> sort(|a, b| a - b)

cluster_ids |> each(|cid|
  println("Cluster " + str(cid) + " top markers:")

  # Get cell indices in this cluster vs all others
  let in_cluster = []
  let out_cluster = []
  let i = 0
  clusters |> each(|c|
    if c == cid then
      in_cluster = concat(in_cluster, [i])
    else
      out_cluster = concat(out_cluster, [i])
    end
    i = i + 1
  )

  # Test each HVG for differential expression
  let markers = hvgs |> head(500) |> map(|g|
    let vals = col_values(log_normalized, g)
    let in_vals = in_cluster |> map(|idx| vals[idx])
    let out_vals = out_cluster |> map(|idx| vals[idx])
    let test = wilcoxon(in_vals, out_vals)
    {
      gene: g,
      pvalue: test.pvalue,
      mean_in: mean(in_vals),
      mean_out: mean(out_vals),
      log_fc: mean(in_vals) - mean(out_vals)
    }
  )

  # Sort by p-value and show top 5
  let top_markers = markers
    |> filter(|m| m.log_fc > 0)  # upregulated in this cluster
    |> sort(|a, b| a.pvalue - b.pvalue)
    |> head(5)

  top_markers |> each(|m|
    println("  " + m.gene + ": logFC=" + str(round(m.log_fc, 2)) +
            " p=" + str(round(m.pvalue, 6)))
  )
)

# Monocytes:   LYZ, S100A8, S100A9, CD14

The Eigenvalue Decomposition in Detail

To fully understand what PCA gives you, let us look at the eigenvalue decomposition directly.

# Compute the correlation matrix of the HVGs
let cor_mat = cor_matrix(scaled_data |> select(hvgs |> head(100)))

# The eigenvalues of the correlation matrix equal the variance
# explained by each PC (when data is scaled)
let eigenvals = eigenvalues(cor_mat)

println("Top 10 eigenvalues of the correlation matrix:")
eigenvals |> head(10) |> each(|e|
  println("  " + str(round(e, 4)))
)

# Eigenvalue > 1 means the PC explains more variance than any single gene
# This is the Kaiser criterion: keep PCs with eigenvalue > 1
let kaiser_count = eigenvals |> filter(|e| e > 1.0) |> len()
println("PCs with eigenvalue > 1 (Kaiser criterion): " + str(kaiser_count))

# Warning: the Kaiser criterion is a rough heuristic
# In genomics with thousands of genes, it often suggests too many PCs
# The elbow plot is more reliable for biological data

Interpretation

What PCA Tells You (and What It Doesn’t)

PCA tells you:

  • How much of the variance is low-dimensional. If 80% of variance is in the first 5 PCs, the data has strong structure.
  • Which genes co-vary. The loadings reveal gene programs — groups of genes that vary together across cells or samples.
  • Whether your data has batch effects. If technical variables correlate with the top PCs, you have a problem.
  • How many distinct groups exist. The number of clear elbows in the scree plot gives a rough estimate.

PCA does NOT tell you:

  • What the groups are. PCA is unsupervised — it does not use labels.
  • Whether nonlinear relationships exist. PCA is strictly linear.
  • Whether small but important subpopulations exist. PCA prioritizes large-variance directions, which may miss rare cell types.

What t-SNE and UMAP Tell You (and What They Don’t)

t-SNE and UMAP tell you:

  • Whether distinct clusters exist. Points that cluster together are genuinely similar in the original space.
  • Local neighborhood structure. Nearby points in the plot are truly nearby in the original space.

They do NOT tell you:

  • Distances between clusters. The spacing is an artifact of the algorithm.
  • Cluster sizes. A large visual cluster may contain fewer cells than a small one.
  • Continuous gradients. Both methods discretize continuous variation into apparent clusters.
  • Statistical significance. Clusters in a UMAP plot are not automatically statistically significant cell types.
# Demonstrate that UMAP distances are not quantitative
# Compute actual distances between cluster centroids in PC space
# vs apparent distances in UMAP space

let centroids_pc = cluster_ids |> map(|cid|
  let mask = clusters |> map(|c| c == cid)
  let pc_vals = pc_embedding |> filter(|row| mask[0])  # simplified
  # In practice: average PC coordinates for cells in this cluster
  { cluster: cid, type: "PC_space" }
)

# Key point: the rank order of inter-cluster distances in PC space
# may not match the rank order in UMAP space
# Two clusters that are far apart in UMAP may be closer in PC space
# than two clusters that appear adjacent in UMAP
println("Warning: UMAP distances are NOT quantitative measures of similarity")
println("Always verify apparent relationships by checking PC space distances")
println("or by computing marker gene overlap between clusters")

Decision

Choosing Your Approach

Given your specific problem, here is the decision tree:

Exploratory analysis of single-cell data: PCA (30 PCs) -> UMAP -> Leiden clustering. This is the standard pipeline. Do not skip PCA.

Quality control of bulk RNA-seq: PCA only. Plot PC1 vs PC2. Color by biological condition AND by all technical variables (batch, lane, extraction date, operator). If any technical variable explains more variance than your biological question, correct the batch effect before proceeding.

Publication figure: Both PCA and UMAP/t-SNE. Show PCA for quantitative claims (variance explained, loadings). Show UMAP/t-SNE for visual clarity. Report the parameters you used.

Discovering gene programs: PCA loadings analysis. The genes with high absolute loadings on a PC form a biological program. Run gene ontology enrichment on these genes to interpret the program.

# Gene program discovery from PC loadings

# Get top 50 genes for PC1 (positive loadings)

  |> filter(|g| g.loading > 0)
  |> head(50)
  |> map(|g| g.gene)

# Get top 50 genes for PC1 (negative loadings)

  |> filter(|g| g.loading < 0)
  |> head(50)
  |> map(|g| g.gene)

println("PC1 positive gene program (myeloid):")
println("  " + str(pc1_pos |> head(10)))

println("PC1 negative gene program (lymphoid):")
println("  " + str(pc1_neg |> head(10)))

# pc1_pos should enrich for innate immunity, phagocytosis
# pc1_neg should enrich for adaptive immunity, T cell receptor signaling

When reporting dimensionality reduction in a paper or presentation:

  1. State the preprocessing: normalization, gene selection, scaling
  2. State the number of PCs used and how you chose it (elbow, cumulative variance threshold, or formal test)
  3. Show the scree plot in supplementary materials
  4. For UMAP/t-SNE: report all parameters (perplexity, n_neighbors, min_dist, random seed, number of iterations)
  5. Do not make distance claims from UMAP/t-SNE plots
  6. Show PC loadings when claiming biological programs

Pitfalls

Pitfall 1: Running t-SNE/UMAP on Raw Gene Expression

Never run t-SNE or UMAP directly on the raw gene expression matrix. The noise in 20,000 genes drowns out the signal. Always reduce to 30-50 PCs first. This denoising step is essential.

# DO NOT do this:
# let bad_tsne = tsne(raw_expression_matrix)

# DO this instead:
# let pca_reduced = pca(scaled_data, n_components: 30)
# let good_tsne = tsne(pca_reduced.components)

println("Always run PCA before t-SNE or UMAP")
println("PCA denoises the data by discarding low-variance components")
println("Without this step, nonlinear methods amplify noise")

Pitfall 2: Over-Interpreting t-SNE Cluster Separation

A common mistake is to claim that two cell types are “highly distinct” because they appear well-separated in a t-SNE plot. The visual separation is an artifact of the algorithm’s optimization. Two groups with 10% gene expression overlap can look as separated as two groups with 0% overlap.

Pitfall 3: Ignoring the Elbow Plot

Choosing too few PCs discards biological signal. Choosing too many includes noise dimensions that degrade clustering. Spend time on the scree plot.

# How the number of PCs affects clustering

let pc_counts = [5, 10, 15, 20, 30, 50]
pc_counts |> each(|n|
  let embedding = final_pca.components |> map(|row| row |> head(n))
  let cl = leiden(embedding, resolution: 1.0)
  let n_clusters = len(unique(cl))
  println("PCs=" + str(n) + " -> " + str(n_clusters) + " clusters")
)

# Too few PCs (5): may merge distinct cell types
# Too many PCs (50): may split cell types by noise
# Sweet spot: where the scree plot elbows (typically 10-30 for single-cell)

Pitfall 4: Not Checking for Confounders in PCA

If PC1 correlates with your variable of interest AND with a technical confounder, you cannot tell which one is driving the pattern. Always test PCs against all known technical variables.

# Check all PCs against technical variables
let technical_vars = ["sequencing_batch", "library_prep_date",
                      "rna_integrity_number", "percent_mito"]

println("Testing PCs against technical variables:")
technical_vars |> each(|var|
  # For continuous variables, use correlation
  # For categorical variables, use ANOVA
  println("  " + var + ": [test result here]")
)

Pitfall 5: Treating UMAP/t-SNE as Deterministic

Both t-SNE and UMAP are stochastic algorithms. Running them twice with different random seeds produces different layouts. If a biological claim depends on the specific layout of a UMAP plot, it is not robust.

# Demonstrate stochasticity
let umap_run1 = umap(pc_input, n_neighbors: 15, min_dist: 0.1)
let umap_run2 = umap(pc_input, n_neighbors: 15, min_dist: 0.1)

# The overall cluster structure should be the same
# But the rotation, reflection, and exact positions will differ
# If your conclusion changes between runs, it is not robust
println("UMAP is stochastic — always verify that conclusions are")
println("consistent across multiple random seeds")

Pitfall 6: PCA on Non-Centered Data

PCA assumes the data is centered (mean-subtracted). If you skip centering, PC1 will point toward the mean of the data, wasting your most informative component on a trivial direction.

# Centering matters
let uncentered = log_normalized |> select(hvgs)
let centered = normalize(uncentered, "zscore")

let pca_uncentered = pca(uncentered, n_components: 5)
let pca_centered = pca(centered, n_components: 5)

# Compare variance explained
println("Uncentered PCA — PC1 variance: " +
        str(round(100.0 * pca_uncentered.variance[0] / sum(pca_uncentered.variance), 1)) + "%")
println("Centered PCA — PC1 variance: " +
        str(round(100.0 * pca_centered.variance[0] / sum(pca_centered.variance), 1)) + "%")

# The uncentered PC1 will explain a huge amount of variance
# but it is trivially pointing toward the mean expression level
# The centered PC1 captures actual biological variation

Where Next?

Dimensionality reduction opens the door to several downstream analyses:

Chapter 14: Clustering & Classification — Once you have a PCA embedding, you can cluster cells or samples into groups. Leiden clustering on PCA coordinates is the standard approach in single-cell genomics. K-means on PCA is used for tumor subtyping.

Trajectory inference — If your cells represent a developmental process (stem cell differentiation, immune activation), dimensionality reduction reveals continuous trajectories rather than discrete clusters. Methods like diffusion maps and pseudotime ordering build on PCA/UMAP embeddings.

Data integration — When combining datasets from different labs, technologies, or species, PCA-based methods like Harmony, Scanorama, and BBKNN align the PC spaces across datasets to enable joint analysis.

Gene regulatory network inference — The co-variation structure captured by PCA can be used to infer which transcription factors regulate which genes. Genes with similar PC loading profiles are likely co-regulated.

Transfer learning — A PCA model trained on a large reference dataset (like the Human Cell Atlas) can be used to project new datasets into the same coordinate system. This enables automated cell type annotation by finding the nearest reference cell type in PC space.

# Example: project a new dataset into the PBMC reference PCA space
# This is the basis of automated cell type annotation

# Suppose we have a new sample
let new_sample = read_tsv("new_pbmc_sample.tsv")

# Normalize the same way as the reference
let new_normalized = normalize(new_sample |> select(hvgs), "zscore")

# Project into reference PC space using the saved loadings
# new_PCs = new_data %*% reference_loadings
let new_projected = new_normalized |> map(|row|
  let pc_coords = []
  let pc_idx = 0
  while pc_idx < n_pcs do
    let score = hvgs |> map(|g|
      let gene_idx = 0
      row[g] * pca_result.loadings[gene_idx][pc_idx]
    ) |> sum()
    pc_coords = concat(pc_coords, [score])
    pc_idx = pc_idx + 1
  end
  pc_coords
)

# Now new_projected is in the same coordinate system as the reference
# You can classify cells by nearest-neighbor in this space
println("Projected " + str(len(new_projected)) + " new cells into reference PC space")
println("Ready for automated cell type annotation")
