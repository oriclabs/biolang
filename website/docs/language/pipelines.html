Pipelines

Bioinformatics workflows are rarely a single step. A typical analysis chains together quality control, alignment, variant calling, filtering, and annotation into a directed acyclic graph of stages. BioLang provides first-class pipeline blocks that make multi-stage workflows explicit, reproducible, and parallel-aware.

Pipeline Blocks

A pipeline block declares a named workflow composed of stage blocks. Stages execute sequentially by default, and each stage can reference the output of any previous stage. 

pipeline fastq_qc {
  stage raw_stats {
    read_fastq("sample_R1.fastq.gz")
      |> filter(|r| mean_phred(r.quality) >= 20)
      |> collect()
      |> len()
  }

  stage filter {
    read_fastq("sample_R1.fastq.gz")
      |> filter(|r| mean_phred(r.quality) >= 30)
      |> write_fastq("sample_R1.filtered.fastq.gz")
  }

  stage filtered_stats {
    read_fastq("sample_R1.filtered.fastq.gz")
      |> collect()
      |> len()
  }
}

When you invoke fastq_qc, BioLang runs raw_stats, then filter, then filtered_stats in order. The return value of the pipeline is the result of the last stage.

Passing Data Between Stages

Stages can capture the result of a previous stage by name. The stage name becomes a binding available to all subsequent stages. 

pipeline align_and_sort {
  stage alignment {
    let ref = "GRCh38.fa"
    let r1 = "tumor_R1.fastq.gz"
    let r2 = "tumor_R2.fastq.gz"
    shell("bwa-mem2 mem -t 16 " + ref + " " + r1 + " " + r2 + " > aligned.sam")
    "aligned.sam"
  }

  stage sorted {
    # alignment is the output path from the previous stage
    shell("samtools sort -@ 8 " + alignment + " -o sorted.bam")
    "sorted.bam"
  }

  stage indexed {
    shell("samtools index " + sorted)
    sorted  # return the sorted BAM path
  }
}

Each stage name resolves to whatever its block evaluated to. This creates an implicit dependency chain without manual wiring.

Stage Dependencies and DAG Execution

BioLang infers a dependency graph from which stage names appear in each block. Independent stages run concurrently when the runtime detects no data dependencies. 

pipeline variant_qc {
  stage aligned {
    shell("bwa-mem2 mem -t 16 GRCh38.fa sample_R1.fq.gz sample_R2.fq.gz | samtools sort -@ 8 -o aligned.bam")
    "aligned.bam"
  }

  # These two stages depend only on aligned, not on each other.
  # BioLang runs them concurrently.
  stage depth_check {
    let bam_reads = read_bam(aligned) |> collect()
    let mapped = bam_reads |> filter(|r| r.is_mapped) |> len()
    mapped
  }

  stage flag_stats {
    let bam_reads = read_bam(aligned) |> collect()
    flagstat(bam_reads)
  }

  # This stage depends on both depth_check and flag_stats
  stage report {
    {
      mapped_reads: depth_check,
      flagstat: flag_stats,
      sample: "sample_01",
    }
  }
}

The runtime builds a DAG: aligned has no dependencies, depth_check and flag_stats both depend on aligned (and thus run in parallel once it finishes), and report depends on both.

Parallel Execution

When you need to process multiple samples, parallel for fans out work across available cores. BioLang schedules iterations concurrently, respecting system resources. 

let samples = [
  {id: "tumor_01", r1: "t01_R1.fq.gz", r2: "t01_R2.fq.gz"},
  {id: "tumor_02", r1: "t02_R1.fq.gz", r2: "t02_R2.fq.gz"},
  {id: "tumor_03", r1: "t03_R1.fq.gz", r2: "t03_R2.fq.gz"},
  {id: "normal_01", r1: "n01_R1.fq.gz", r2: "n01_R2.fq.gz"},
]

parallel for sample in samples {
  let bam = sample.id + ".sorted.bam"
  shell("bwa-mem2 mem -t 4 GRCh38.fa " + sample.r1 + " " + sample.r2
        + " | samtools sort -@ 4 -o " + bam)
  shell("samtools index " + bam)
  {id: sample.id, bam: bam}
}

The result is a list of records, one per sample, collected in the order the iterations complete.

Defer for Cleanup

Intermediate files pile up fast. defer registers a cleanup action that runs when the pipeline completes, whether it succeeds or fails. Multiple defer blocks execute in reverse order (LIFO). 

pipeline trimmed_alignment {
  stage trim {
    let trimmed_r1 = "tmp/trimmed_R1.fq.gz"
    let trimmed_r2 = "tmp/trimmed_R2.fq.gz"
    trim_adapters("sample_R1.fq.gz", "sample_R2.fq.gz",
                  out_r1: trimmed_r1, out_r2: trimmed_r2,
                  adapter: "AGATCGGAAGAG")
    defer { remove(trimmed_r1); remove(trimmed_r2) }
    {r1: trimmed_r1, r2: trimmed_r2}
  }

  stage align {
    let bam = "aligned.bam"
    shell("bwa-mem2 mem -t 16 GRCh38.fa " + trim.r1 + " " + trim.r2
          + " | samtools sort -@ 8 -o " + bam)
    let unsorted = "aligned.unsorted.bam"
    defer { remove(unsorted) }
    bam
  }
}

When trimmed_alignment finishes, the deferred blocks fire: the unsorted BAM is removed first, then the trimmed FASTQs. Cleanup happens even if a stage fails.

Parameterized Pipelines (Templates)

A pipeline can accept parameters, turning it into a reusable template. When you declare pipeline name(param1, param2) { ... }, BioLang defines a callable function instead of executing immediately. You invoke it like any other function. 

# Define a reusable alignment template
pipeline align_sample(sample_id, r1, r2, reference) {
  stage aligned {
    shell("bwa-mem2 mem -t 16 " + reference + " " + r1 + " " + r2
          + " | samtools sort -@ 8 -o " + sample_id + ".sorted.bam")
    sample_id + ".sorted.bam"
  }

  stage indexed {
    shell("samtools index " + aligned)
    aligned
  }
}

# Call with different inputs — same pipeline, different data
let tumor = align_sample("tumor", "tumor_R1.fq.gz", "tumor_R2.fq.gz", "GRCh38.fa")
let normal = align_sample("normal", "normal_R1.fq.gz", "normal_R2.fq.gz", "GRCh38.fa")
print("Tumor BAM: " + tumor)
print("Normal BAM: " + normal)

Parameterized pipelines compose naturally. You can call one template from inside another, loop over a sample list, or pass configuration records as parameters. 

# Process a whole cohort with the same template
let samples = [
  {id: "S1", r1: "S1_R1.fq.gz", r2: "S1_R2.fq.gz"},
  {id: "S2", r1: "S2_R1.fq.gz", r2: "S2_R2.fq.gz"},
  {id: "S3", r1: "S3_R1.fq.gz", r2: "S3_R2.fq.gz"},
]

let bams = samples |> map(|s| align_sample(s.id, s.r1, s.r2, "GRCh38.fa"))
print("Aligned " + str(len(bams)) + " samples")

How it works:

  • pipeline name { ... } (no params) — executes immediately, result bound to name
  • pipeline name(params) { ... } — defines a callable function; nothing runs until you call it
  • The parameter list uses the same syntax as function parameters

Pipeline Composition

Pipelines are values. You can call one pipeline from within another or store its result for downstream use. 

pipeline align_one(sample) {
  stage bam {
    let out = sample.id + ".sorted.bam"
    shell("bwa-mem2 mem -t 8 GRCh38.fa " + sample.r1 + " " + sample.r2
          + " | samtools sort -@ 4 -o " + out)
    out
  }
  stage index {
    shell("samtools index " + bam)
    bam
  }
}

pipeline somatic_pair(tumor, normal) {
  stage tumor_bam { align_one(tumor) }
  stage normal_bam { align_one(normal) }

  stage call {
    shell("gatk Mutect2 -R GRCh38.fa -I " + tumor_bam
          + " -I " + normal_bam + " -normal " + normal.id
          + " -O somatic.vcf.gz")
    "somatic.vcf.gz"
  }
}

# Run the composed pipeline
let result = somatic_pair(
  {id: "tumor", r1: "tumor_R1.fq.gz", r2: "tumor_R2.fq.gz"},
  {id: "normal", r1: "normal_R1.fq.gz", r2: "normal_R2.fq.gz"}
)
print("Somatic variants written to: " + result)

Introspection Builtins

Two builtins help you inspect and optimize pipelines:

pipeline_steps(pipeline_record) Record → List

Extracts the list of stage names and metadata from a pipeline record. 

let steps = pipeline_steps(fastq_qc)
# [{name: "raw_stats"}, {name: "filter"}, {name: "filtered_stats"}]
pipe_fuse(list, op1, op2, ...) List, Fns → List

Fuses multiple pipe operations into a single pass over the data for efficiency. Instead of creating intermediate lists between each operation, pipe_fuse applies all operations in one traversal. 

# Without fusion: creates intermediate lists
let result = reads
  |> filter(|r| mean_phred(r.quality) >= 30)
  |> map(|r| gc_content(r.seq))
  |> filter(|gc| gc >= 0.3 and gc <= 0.7)

# With fusion: single pass
let result = pipe_fuse(reads,
  |r| mean_phred(r.quality) >= 30,
  |r| gc_content(r.seq)
)

Example: QC-to-Variants Pipeline

A complete workflow from raw reads to filtered variants using BioLang builtins:

pipeline qc_to_variants {
  stage qc {
    let reads = read_fastq("sample_R1.fq.gz") |> collect()
    let total = len(reads)
    let passing = reads |> filter(|r| mean_phred(r.quality) >= 25) |> len()
    print("QC: " + str(passing) + "/" + str(total) + " reads pass Q25")

    # Write filtered reads
    reads
      |> filter(|r| mean_phred(r.quality) >= 25)
      |> write_fastq("sample_R1.filtered.fq.gz")
  }

  stage align {
    shell("bwa-mem2 mem -t 16 GRCh38.fa sample_R1.filtered.fq.gz"
          + " | samtools sort -@ 8 -o aligned.bam")
    shell("samtools index aligned.bam")
    "aligned.bam"
  }

  stage call {
    shell("gatk HaplotypeCaller -R GRCh38.fa -I " + align
          + " -O raw.vcf.gz")
    "raw.vcf.gz"
  }

  stage filter_variants {
    let variants = read_vcf(call)
    let snps = variants |> filter(|v| v.is_snp)
    let indels = variants |> filter(|v| v.is_indel)

    print("SNPs: " + str(len(snps)) + ", Indels: " + str(len(indels)))
    print("Ti/Tv: " + str(tstv_ratio(snps)))

    variants
      |> filter(|v| v.quality >= 30.0)
      |> write_vcf("filtered.vcf.gz")
  }
}

# Run the full pipeline
let output = qc_to_variants
print("Pipeline complete: " + output)

Pipe Binding with |> into

Inside pipeline stages and regular code, you can use |> into to bind intermediate pipe results to a name without wrapping the chain in let x = .... The syntax expr |> into name is equivalent to let name = expr, but reads left-to-right. 

pipeline annotate {
  stage load {
    read_vcf("calls.vcf.gz")
      |> filter(|v| v.qual >= 30)
      |> into passing

    # passing is now bound and can be reused
    print(f"Passing variants: {len(passing)}")

    passing
      |> filter(|v| v.is_snp)
      |> into snps

    passing
      |> filter(|v| v.is_indel)
      |> into indels

    {snps: snps, indels: indels, total: len(passing)}
  }
}

This is especially useful in stages where you need to reference intermediate results multiple times without breaking the pipeline flow.
